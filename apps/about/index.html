BORK Kernel Architecture
bork.sys.*Public syscall surface — the only interface BARK ever calls
ProcessMgrPIDs, FD tables, signals (SIGTERM/KILL/CHLD/USR1/USR2), job control
VFS RouterPath resolution, mount table, multi-backend dispatch
MemFSIn-RAM inode tree — /tmp, /dev, /lib, /mnt

ProcFSSynthetic /proc — live stats from kernel state
ZipFSZIP archive backend — mount any .zip as read-only filesystem

NetDevicefetch() bridge at /dev/net — wget/curl via real HTTP
Pipe IPCBidirectional ring buffers for shell pipelines: cmd1 | cmd2 | cmd3
ShellParserPipes, redirects, env vars, quoting, background jobs
What makes this different from just JS

A static GitHub Pages site can run JavaScript but has no persistent storage, no filesystem abstraction, no process model, no real I/O. Every "app" is a blank canvas reinventing the wheel.

SubstrateOS provides a stable platform: apps call bork.sys.* and get a real FS, real persistence, real network, real pipes, real processes. The underlying implementation can be swapped (MemFS → IDBFS → future OPFS) without the app knowing.

The shell pipeline system (cat file | grep foo | wc -l > out.txt) runs across real Pipe objects with backpressure. Output redirection writes to real IDB-backed files. wget fetches real URLs and saves to VFS. Drag a ZIP → it mounts as a filesystem with zero extraction.

This is what GitHub Pages can be if you bring the runtime with you.
